## Аккаунты / Firebase (новое, 2026-07)

Сайт теперь требует входа (почта+пароль или телефон+SMS-код) — весь UI
закрыт `AuthPage` в `src/App.jsx` пока `useAuthStore` не отдаст
пользователя. Проект Firebase: `productivityapp-92a92`, конфиг лежит в
`.env.local` (гитигнорится через `*.local`, при новом клоне репозитория
нужно создать заново из Firebase Console → Project settings → Web app).

**Данные разделов теперь в Firestore, не в localStorage.** Паттерн для
любого нового `persist`-стора:
```js
persist(fn, {
  name: 'productivity-hub-<key>',
  storage: createFirestoreStorage('productivity-hub-<key>'),
  skipHydration: true,
  partialize: (state) => ({ /* только данные, НЕ экшены */ }),
})
```
- `src/firebase/firestoreStorage.js` — сам адаптер, пишет в
  `users/{uid}/appData/{key}`.
- `partialize` обязателен — без него zustand кладёт в стейт и функции-
  экшены, а Firestore падает с `Unsupported field value: a function`
  (localStorage эту ошибку скрывал через `JSON.stringify`, поэтому баг
  не был виден раньше — наступили на эти грабли один раз, задокументировано).
- `skipHydration: true` — рехидратация запускается вручную из
  `src/store/rehydrateStores.js` (`rehydrateAllPersistedStores()`) после
  входа, а `resetAllStores()` — при выходе, чтобы данные одного аккаунта
  не утекали в интерфейс другого до того, как подгрузятся его собственные.
- `src/firebase/currentUid.js` — намеренно НЕ `useAuthStore`: если
  `firestoreStorage.js` импортирует хук стора, а `useAuthStore` через
  `rehydrateStores.js` импортирует все сторы — получается цикл модулей и
  `ReferenceError: Cannot access '...' before initialization`. Этот
  отдельный модуль-лист разрывает цикл.

**Друзья и соревнования** (`useFriendsStore`, `useCompetitionsStore`,
кнопка добавления — в Настройках, не на вкладке Трофеи):
- `friendRequests/{fromUid}_{toUid}` — заявка И одновременно запись о
  дружбе после `status: 'accepted'`. Отдельного списка друзей на
  пользователя нет специально: mutual-write в чужую подколлекцию без
  бэкенда не работает (проверено на практике, см. историю) — обе стороны
  должны сами писать в СВОЮ подколлекцию, а второй пользователь может не
  быть в сети в момент принятия заявки.
- `userDirectory/{email} -> {uid}` — чтобы искать пользователя по почте
  (Firebase Auth сам такого API не даёт).
- `competitions/{id}` + `participantHours/{uid}` — часы каждого участника
  в своём документе (простое правило безопасности: пишешь только свой uid),
  раз в минуту подтягиваются из `useFocusHistoryStore` и пушатся в
  Firestore, пока соревнование активно.
- Правила — `firestore.rules` в корне репозитория, публикуются вручную в
  Firebase Console → Firestore → Rules (CLI не используется). **Если
  меняешь `firestore.rules` в коде — не забудь попросить пользователя
  переопубликовать** — тестировали, легко забыть и словить
  `permission-denied`, которое выглядит как баг в коде, а на деле просто
  старые правила.

## Квота Firestore (сессия 2026-07) — почему упирались и что починили

Бесплатный тариф Spark: 50k чтений / 20k записей / 20k удалений в день,
сбрасывается около полуночи по тихоокеанскому времени. Один разработчик
умудрялся выедать весь дневной лимит в одиночку — расследовали, откуда.

**Главный виновник (уже исправлен):** `useFocusHistoryStore.addSecond()`
вызывался на каждый тик таймера (раз в секунду, см.
`useFocusTimerStore.js` `_tick()`) и раньше писал в Firestore на каждый
вызов — ~3600 записей в час активного фокуса, ~1500 записей за одну
25-минутную сессию Помодоро. Это единолично убивало квоту меньше чем за
6 часов суммарного использования таймера — и это реальная нагрузка от
КАЖДОГО пользователя, не артефакт разработки.
- **Фикс:** вынес троттлинг из `useFocusTimerStore.js` в переиспользуемый
  `src/firebase/throttledStorage.js` (`createThrottledStorage`) — большинство
  записей теперь просто обновляют значение в памяти, а реальный `setDoc`
  улетает не чаще раза в `HISTORY_SAVE_INTERVAL_MS` (20 сек), плюс
  принудительно на `visibilitychange`/`pagehide`, чтобы не терять больше
  нескольких секунд прогресса при закрытии вкладки. Тот же паттерн уже
  был в `useFocusTimerStore.js` (там это было исправлено раньше — теперь
  оба стора используют один и тот же общий хелпер вместо дублирования).
  **Функционал не пострадал** — сохраняется всё то же самое, просто реже.

**Второй виновник (тоже исправлен):** `useFriendsStore`/`useCompetitionsStore`
держали два постоянных `onSnapshot`-слушателя + запись раз в минуту
(пока активно соревнование) — раньше это запускалось `start()`
автоматически **при каждом входе в аккаунт**, то есть каждый
залогиненный пользователь платил эту цену всю сессию, даже если ни разу
не открывал панель друзей.
- **Фикс:** `start()`/`stop()` перенесены из `useAuthStore.js` (при
  логине/логауте) в `useEffect` внутри `src/features/social/FriendsPanel.jsx`
  (при монтировании/размонтировании) — эта панель и так рендерится
  только когда развёрнут аккордеон "Фокус" в Настройках → Section
  Settings, так что слушатели теперь живут ровно пока панель открыта.
  `useAuthStore.js` при выходе всё ещё на всякий случай вызывает `stop()`
  (защитный код на случай разлогина прямо во время открытой панели).
  **Побочный эффект, о котором стоит знать:** если соревнование
  заканчивается, а пользователь ни разу не открыл Настройки за эти 10
  дней — трофей за победу засчитается не ровно в момент истечения
  таймера, а при следующем открытии панели друзей (`_maybeFinish`
  проверяется на пуше часов, а пуш происходит только пока слушатель
  открыт). На практике не страшно — это не push-уведомление, пользователь
  и так узнаёт результат, когда заходит проверить.

**Раунд 2 оптимизации (та же сессия, "максимум с сохранением функционала"):**
проверил `useFocusRankStore.reconcile()` и `useTrophiesStore.checkStreakMilestones`
— оба уже идемпотентны (ранний `return` без `set()`, если ничего не
изменилось), лишних записей при каждом заходе на `/focus` не создают,
трогать не пришлось. Дальше выжал интервалы троттлинга:
- `SESSION_SAVE_INTERVAL_MS` (`useFocusTimerStore.js`) и
  `HISTORY_SAVE_INTERVAL_MS` (`useFocusHistoryStore.js`): **20 сек → 60 сек**
  (3x меньше записей за минуту активного таймера). Риск — до минуты
  прогресса потеряется при истинном креше/отключении питания (не при
  обычном закрытии вкладки, там мгновенный `flush()` на
  `visibilitychange`/`pagehide` уже есть) — на статистику/ранг это
  не влияет заметно.
- `HOURS_PUSH_INTERVAL_MS` (`useCompetitionsStore.js`): **60 сек → 5 мин**
  (5x меньше записей+чтений, но только пока открыта панель друзей и
  идёт соревнование — оно длится 10 дней, отставание в 5 минут ничего
  не решает).

**Не трогали (уже эффективно или низкая отдача от риска):**
- Рехидрация 7 сторов при логине (`rehydrateAllPersistedStores()`) — по
  одному `getDoc` на стор, разово при входе. Для реального пользователя
  это не проблема (не перезагружает страницу по 20 раз в час, как
  разработчик во время активной разработки).
- `ensureUserProfile()` в `useAuthStore.js` пишет в `userDirectory` при
  каждом `onAuthStateChanged` (то есть при каждом открытии вкладки
  заново) — рассматривали кэшировать локально, чтобы не писать повторно
  для того же uid, но цена там мизерная (1-2 записи на открытие
  вкладки, не на тик/минуту) — оставили как есть, не тот масштаб
  проблемы.

  Хэш-навигация (`#/focus`) всегда бьёт в единственный `index.html`,
  проблема решена полностью. Если когда-нибудь будет настоящий сервер с
  rewrite — можно вернуть `BrowserRouter`, но сейчас трогать не надо.
- Есть готовый `.github/workflows/deploy.yml` (сборка + деплой через
  Actions) на случай, если пользователь всё-таки настроит `git push` —

## UI-полировка раздела "Фокус" (сессия 2026-07, после Firebase)

- **Плавающий виджет таймера** (`FloatingTimerWidget.jsx`, рендерится в
  `App.jsx` Layout, всегда смонтирован): показывается когда таймер идёт,
  а полноэкранный вид закрыт (`useFocusUiStore.timerViewOpen`).
  Перетаскивается (pointer events), двойной клик открывает полный экран
  через `useFocusUiStore.selectedProjectId` (НЕ через local state — иначе
  `navigate('/focus')` не срабатывает, если уже на /focus). Есть кнопка
  Document Picture-in-Picture (`documentPictureInPicture.requestWindow`)
  — настоящее окно поверх ВСЕХ приложений, не только браузера. Только
  Chrome/Edge/Opera, нужен настоящий клик пользователя (не тестируется
  автоматизацией). Содержимое PiP-окна — отдельная простая разметка
  (`.focus-timer-pip`), не переиспользует стили плавающего виджета.
- **Сайдбар**: сворачивается свайпом влево (touch) ИЛИ кнопкой прямо в
  самой панели (`.sidebar__collapse-btn`, как в Claude) — состояние в
  `useSidebarStore`. Когда свёрнут — гамбургер в TopBar открывает обратно.
  **С 2026-07 персистится** (обновление "запоминай мои переключатели") —
  но **в `localStorage`, не в Firestore**: это layout-настройка конкретного
  устройства/окна (на телефоне и десктопе разумные значения могут
  отличаться), плюс нужна мгновенно при первом рендере без ожидания
  асинхронного чтения — поэтому обычный `persist` без
  `createFirestoreStorage`, нулевая стоимость по квоте. Тот же паттерн
  (`src/store/useUiPreferencesStore.js`, ключ `collapsed: { [key]: bool }`)
  используется для сворачиваемых секций на странице Задач
  (`tasks.unsorted`/`tasks.tomorrow`/`tasks.week`/`tasks.future`) и для
  папки "Завершённые проекты" в `ProjectList.jsx`
  (`focus.completedProjectsFolder`) — если появится ещё один сворачиваемый
  блок, который должен запоминать себя, добавляй новый ключ в этот же
  стор, не создавай новый.
- **Проекты**: список — снова обычные строки на всю ширину (`.project-list`
  flex-column с общей рамкой), НЕ grid-сетка карточек — пробовали grid,
  пользователю не понравилось, откатили. Переименование — карандаш
  открывает инлайн-инпут (`useFocusStore.renameProject`). "Циклы"
  переименованы в "сессии" везде в UI (сам стор/поле `cycles` не
  переименовывали — только текст).
- **Таймер**: кольцо 360px, линия 14px, цифры 3.8rem, весь `.timer-view`
  до 960px (было 720, было 560 — пользователь просил меньше пустого места
  по бокам). На фазе отдыха кольцо и кнопка "+5 минут" ИСЧЕЗАЮТ, вместо
  них — просто цифры + текстовая рекомендация отдохнуть (`.timer-rest`).
  "Сегодня в фокусе" — цветной бейдж, не тусклый текст. Фоновые звуки
  свёрнуты в раскрывающуюся кнопку (`.sound-picker__toggle`), не грид
  всегда открытых кнопок.
- **Рингтоны при смене фазы**: `src/audio/longRingtones.js` — 10 тонов на
  начало работы + 10 на начало отдыха, каждый ≥30 сек (генерируется
  циклом короткого паттерна до потолка `MIN_DURATION_S=32`). Выбираются в
  Настройках → Фокус (`RingtonePicker.jsx`), хранится в
  `useSettingsStore.workStartRingtoneId`/`restStartRingtoneId` (`null` =
  случайный). Пулы звучат РАЗНО по замыслу: work-start — без фильтра,
  резкая атака, sine/triangle/square/sawtooth вперемешку, быстрее;
  rest-start — все с `filterCutoff` (lowpass, приглушённее), у части
  вибрато, только sine/triangle, медленнее. Если пользователь снова
  скажет "звучит одинаково" — крутить `filterCutoff`/`vibrato`/
  `harmonyInterval`/`waveform` дальше, не просто менять ноты. Старый
  короткий `ringtones.js` теперь содержит ТОЛЬКО `playReminderTone`
  (напоминание "вы ещё здесь?") — остальное дублировало новую систему,
  удалено.
- **Прерывание звука другим приложением/записью**: технически
  невозможно для веба (браузеры не дают сайту видеть чужой аудио/микрофон
  статус). Реализовано единственное реальное — реакция на
  `AudioContext.onstatechange`/`suspended` (ловит, например, входящий
  звонок на мобильном) в `src/audio/audioContext.js`. Если пользователь
  снова попросит "детект другого приложения" — объяснять ограничение
  заново, не пытаться обойти.

  зацикливается.
- **Статистика**: годовой календарь-теплокарта, серии (streak),
  разбивка времени по проектам (день/неделя/месяц/год), клик по дню —
  детали.
- **Ранг**: 25 рангов (Бронза→Завоеватель, по 3 подуровня + без
  подуровней у Завоевателя), очки = минуты фокуса минус штраф за
  пропущенные дни (100/день, не ниже 0). Своя иконка-силуэт на тир
  (`RankIcons.jsx`), цвет меняется по подуровню (HSL-сдвиг, не
  чёрным/белым). Встроен в дашборд "Проекты", не отдельная вкладка.
- **Трофеи**: 3 вида орденов, по ценности — серии дней (10 порогов,
  персистентны, дублируются), соревнования (10 дней, часы соперника
  синхронизируются через Firestore в реальном времени — см. раздел
  "Аккаунты / Firebase" выше), ранговые (живые, исчезают при понижении
  ранга). Вкладка "Трофеи" показывает ТОЛЬКО сами трофеи — форма запуска
  соревнования и список друзей живут в Настройках (`FriendsPanel.jsx`).

  со свежими после `window.location.reload()` — не доверяй одному чтению

## Раздел "Задачи и цели" — MVP (сессия 2026-07)

Портировали в React логику из отдельного HTML/CSS/JS-прототипа
(`~/Desktop/task-manager/index.html` в другом репозитории), где эта
концепция была сначала обкатана и провалидирована с пользователем. MVP
сознательно урезан относительно прототипа — см. ниже, что отложено.

- **Стор**: `src/store/useTasksStore.js` — `tasks[]` + `projects[]`,
  обычный `persist`+Firestore паттерн (`productivity-hub-tasks`),
  добавлен в `PERSISTED_STORES` в `rehydrateStores.js`.
- **Компоненты**: `src/features/tasks/` — `taskLogic.js` (чистая логика
  классификации по датам и парсинг `чч:мм`, без React), `TaskCard.jsx`,
  `TaskSection.jsx` (сворачиваемая секция), `TaskFormModal.jsx` (общая
  модалка создания И редактирования — при редактировании требование
  "описание и проект обязательны" снимается, чтобы можно было свободно
  донастроить быстро добавленную задачу), `ProjectsTab.jsx`.
  `src/pages/TasksGoalsPage.jsx` — точка сборки (вкладки Задачи/Проекты).
- **Классификация** (`classifyTask` в `taskLogic.js`): Сегодня /
  Неразобранные (нет даты и не "когда-нибудь") / Завтра / На этой неделе
  / В будущем (дальше недели ИЛИ явно отмечено "когда-нибудь"). Готово
  задачи из "Сегодня" не исчезают сразу, а остаются зачёркнутыми в
  секции до конца дня (см. `doneToday` в `TasksGoalsPage.jsx`) — умышленно,
  для наглядности "сколько сегодня сделано".
- **Цвета — токены проекта, не прототипа**: приоритет P1/P2/P3/без —
  `--danger`/`--warning`/`--accent`/`--text-muted`, секция "Неразобранные"
  подсвечена `--warning` (в прототипе была своя красно-оранжево-жёлтая
  палитра и оранжевый акцент — здесь заменено на существующие переменные
  темы, см. `:root` в `src/styles/index.css`).
- **Экспорт**: `collectSectionData.js` теперь читает `useTasksStore` для
  `section.id === 'tasks'` и отдаёт плоские объекты (заголовок, описание,
  проект, приоритет, дедлайн, теги) — экспортёры (`exportToPdf/Excel/Word`)
  не трогали, они уже общие.
- **i18n**: добавлен namespace `tasks` во все три языка (`ru/en/uz.json`),
  ключи синхронизированы (проверено программно, см. историю сессии).
  `sections.tasks.title/description` уже существовали от заглушки —
  переиспользованы для пункта в сайдбаре, не дублировались.

**Осознанно отложено на следующую сессию** (не начинать без запроса
пользователя): drag-and-drop ручная сортировка карточек, повторяющиеся
задачи (ежедневно/по дням недели/еженедельно/ежемесячно/ежегодно),
автоперенос просроченных дат + счётчик переносов, полоса "набрано часов
на сегодня", счётчик "сегодня выполнено", фильтр по тегам, архив
выполненных задач, вечерний ритуал-баннер, общий баннер просроченных
дедлайнов (бейдж просрочки на самой карточке — есть, общего баннера
наверху — нет). У всех этих фич уже есть работающая референсная
реализация в прототипе (`~/Desktop/task-manager/index.html`) — переносить
оттуда, не изобретать заново.

**UI-полировка (сессия 2026-07, после MVP):**
- **Карточка задачи — одна строка по умолчанию** (`TaskCard.jsx`): чекбокс
  + точка приоритета + название (обрезается многоточием, `text-overflow:
  ellipsis`) + кнопка `⋮` (`IconMoreVertical`) + удалить. Описание/бейджи
  (проект/время/дедлайн/теги) скрыты в `.task-card__details`, раскрываются
  локальным `useState` по клику на `⋮` — не персистится (осознанно, это
  временный "peek", не настройка). Клик по названию по-прежнему открывает
  полную модалку редактирования — это отдельное действие от `⋮`.
- **Бейдж-уведомление в углу секции** (`TaskSection.jsx`): красная обводка
  (`--danger`) для Сегодня/Неразобранные, зелёная (`--success`) для
  Завтра/Неделя/Будущее, только если в секции есть карточки. Для Сегодня
  считает **только активные** задачи (`badgeCount` проп) — не путать с
  зачёркнутыми "сделано сегодня", которые тоже рендерятся в этой секции.
  Реализовано `position: absolute` поверх секции — из-за этого `.task-
  section` больше не обрезает `overflow: hidden` (иначе бейдж бы срезался
  по краю), скругление углов сделано вручную на `__header`/`__body`
  (тот же приём, что в `.project-list` у Фокуса).
- **Проекты различаются по цвету** (`getProjectColor()` в `taskLogic.js`):
  при создании проекта (`useTasksStore.addProject`) назначается цвет из
  того же `COLOR_PALETTE`, что уже использует Фокус — циклически по
  порядку создания. Точка с этим цветом показывается в списке вкладки
  "Проекты", в панели выбора проекта модалки и в бейдже проекта на
  раскрытой карточке. **Обратная совместимость**: проекты, созданные до
  этого изменения, не имеют поля `color` — `getProjectColor()` в этом
  случае считает цвет из хеша `id` проекта (стабильно для одного и того
  же проекта, не зависит от позиции в списке).
- **Квота Firestore не тронута этим обновлением** — все три пункта чисто
  UI/производные от уже загруженных данных, ноль новых чтений/записей.

**Как проверялось**: реальный вход через Firebase был недоступен в
сессии портирования (Firestore квота `resource-exhausted` — не связано с
кодом задач, экран зависал на "Загрузка..."). С явного разрешения
пользователя `App.jsx` был временно и обратимо пропатчен (`&& false` на
обеих проверках `authStatus`/`user`), чтобы прогнать страницу вживую в
браузере без входа, затем откат подтверждён через `git diff` (пустой).
Если снова понадобится проверить UI без реального Firebase — тот же
приём, но **обязательно спросить пользователя заново**, это не подразумевается
автоматически на будущее.
